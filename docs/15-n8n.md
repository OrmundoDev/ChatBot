# 15. n8n

## 15.1 Para que serve, em geral

n8n é uma ferramenta de automação visual — permite montar fluxos
("workflows") conectando serviços diferentes através de uma interface
gráfica de nós (*nodes*), sem escrever código para a integração em si (ex:
"quando chegar uma mensagem no WhatsApp, chame esta API, depois responda
com o resultado").

## 15.2 Papel histórico e status atual neste projeto

Na fase inicial do projeto (antes da arquitetura multi-tenant atual), o n8n
era o orquestrador do fluxo de mensagens: Evolution API → n8n → endpoint
`POST /chat` do backend → n8n → Evolution API → WhatsApp. Esse fluxo ainda
**existe tecnicamente** (o endpoint `/chat` em `main.py` continua ativo,
propositalmente mantido por compatibilidade — ver
[capítulo 16](./16-api.md)), mas **não é o caminho usado em produção hoje**.

Desde a introdução dos webhooks diretos (`/webhooks/whatsapp_cloud` e
`/webhooks/evolution`), o backend recebe e responde mensagens diretamente,
sem depender do n8n. O container do n8n continua rodando na VPS (definido
no `docker-compose.yml`, usando o Postgres como seu banco, no schema
`n8n`), mas **fora do caminho de produção** — é infraestrutura legada
mantida ativa, não removida, mas não crítica para o funcionamento atual do
chatbot.

## 15.3 Acessar o n8n

```
http://127.0.0.1:5678
```
Só acessível de dentro da VPS (porta não exposta publicamente) — para
acessar da sua máquina, use um túnel SSH:
```bash
ssh -L 5678:127.0.0.1:5678 deploy@SEU_IP_VPS
```
E abra `http://127.0.0.1:5678` no navegador local.

## 15.4 Importação e exportação de workflows

Pela interface do n8n: um workflow pode ser exportado como JSON (menu do
workflow → Download) e reimportado da mesma forma em outra instância. Não
há workflows documentados como parte ativa do fluxo de produção deste
projeto no momento — não há nada a exportar/versionar como crítico hoje.

## 15.5 Uma nota importante sobre a tabela `n8n` no banco

O n8n cria e gerencia suas próprias tabelas dentro do mesmo banco Postgres,
mas num **schema separado** (`n8n`), diferente do schema `public` usado
pelo restante do sistema (`companies`, `agents`, etc.). Isso já causou
confusão real durante o desenvolvimento: uma consulta em
`information_schema.columns` sem filtrar por schema retornou colunas de
**duas tabelas diferentes chamadas `agents`** — a do backend (`public.agents`)
e uma tabela interna do n8n (`n8n.agents`, com colunas em `camelCase`,
padrão característico de ferramentas baseadas em TypeORM/Prisma). Ao
escrever qualquer SQL neste projeto, é uma boa prática **sempre qualificar
o schema explicitamente** (`public.agents`, não só `agents`) para eliminar
esse tipo de ambiguidade, mesmo que o `search_path` da conexão normalmente
resolva para `public` por padrão.

## 15.6 Quando reconsiderar o uso do n8n

Se no futuro for necessário automatizar algo que não envolva diretamente o
fluxo de mensagens do chatbot (ex: notificações internas, integrações com
outras ferramentas de terceiros que não têm biblioteca Python pronta), o
n8n é uma opção já disponível na infraestrutura, sem custo de setup
adicional. Não é recomendado reintroduzi-lo no caminho principal de
mensagens (WhatsApp → resposta), já que os webhooks diretos são mais
simples de depurar (um serviço a menos na cadeia) e já estão validados em
produção.
